# 営業エントランス ボタンとライン運用の対応表

## 目的
営業がどこまで進めれば現場の entrance cleaning にライン開始が出るかを説明できる状態にする。
Ticket B の仕様と状態機械を前提とする。

## 対象
- 画面: /entrance/sales
- ボタン: 顧客 申込書を作成 契約を締結 店舗査定 予定

## ボタン一覧とライン視点の対応
| ボタン | 役割 何を確定するか | 生成 更新されるデータ | ラインとの関係 |
|---|---|---|---|
| 顧客 | 顧客情報の確定 | 顧客 店舗 ブランドの紐づけ情報 | 間接 顧客 店舗が無いと予定作成に進めないため |
| 申込書を作成 | 申込書の作成開始 | 申込書 依頼情報 | 間接 予定作成の材料になるため |
| 契約を締結 | 契約の確定 | 契約状態 | 間接 予定作成の前後に必要な確定情報 |
| 店舗査定 | 店舗査定の実施 | 店舗査定情報 | 間接 店舗情報の補完に使われる |
| 予定 | 予定の作成確認 | schedules 作成 更新 | 直接 ライン対象の予定が作成されるため |

## 予定作成とライン開始ボタンの出現条件
### 予定作成で生成されるもの
- schedules に予定が作成される
- 予定には scheduled_date scheduled_time store_id status などが保存される

### 現状の実運用での確定点
- 申込書を作成 で依頼内容を確定
- 実運用の確定点は 予定 作成フロー

### 当日の cleaning entrance 表示条件
- Ticket B 仕様のボタン状態判定に従う
- 判定ソース: 当日または最新の line 対象 report の status
- 判定方法: /line/status に worker_id と date を渡して状態を取得
- ボタン表示:
  - in_progress の場合 進行中
  - pending_approval の場合 承認待ち
  - その他の場合 ライン開始

## 予定があるのにライン開始が出ない時のチェックリスト
1) 当日の date が一致しているか
2) worker_id が想定の担当者で一致しているか
3) /line/status の応答 status が想定と一致しているか
4) 予定が line 対象として扱われているか

## 禁止事項
- Ticket B の仕様と状態機械は変更しない
- 画面追加や UI 改修を提案しない
- 現場自由記述を増やさない
- 非営業での AI 誘導は追加しない
- 通常案件を止める仕様にしない
