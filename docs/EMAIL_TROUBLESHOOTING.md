# メールが届かないときの確認ポイント

「メールも届かない」場合、**送信（システムから出すメール）** と **受信（@misesapo.co.jp / @misesapo.app に届くメール）** で原因が違います。両方の確認手順をまとめます。

---

## 1. 送信メールが届かない（システム → 顧客・社内）

アプリや Lambda から送るメール（登録申込書、通知など）の送信元は **`info@misesapo.co.jp`** で、**AWS SES** を使っています。

### 1.1 想定される原因

| 原因 | 説明 |
|------|------|
| SES サンドボックス | 未検証のアドレスには送信のみ可能で、届かないことがある |
| ドメイン未検証 | `misesapo.co.jp` が SES で検証されていない |
| バウンス・苦情 | 過去にバウンスやスパム報告が多いと制限される |
| 送信制限 | サンドボックス解除後も送信枠を超えている |

### 1.2 AWS コンソールで確認すること

1. **SES のリージョン**: Lambda と同じ **ap-northeast-1（東京）** を使っているか確認。
2. **SES →  Identities**:
   - **ドメイン**: `misesapo.co.jp` が「検証済み」か。
   - **メールアドレス**: `info@misesapo.co.jp` を個別に検証している場合は、そのアドレスも「検証済み」か。
3. **SES → Account dashboard**:
   - **Sending status**: サンドボックスなら「Sandbox」、本番送信可能なら「Production」。
   - **Reputation**: バウンス率・苦情率が高くないか。
4. **SES → Suppression list**: バウンスや苦情で登録されたアドレスに送っていないか。

### 1.3 AWS CLI で確認する

```bash
REGION=ap-northeast-1

# ドメイン misesapo.co.jp の検証状態
aws ses get-identity-verification-attributes \
  --identities "misesapo.co.jp" \
  --region "$REGION" \
  --query 'VerificationAttributes.misesapo.co.jp.VerificationStatus' \
  --output text

# 送信元メール info@misesapo.co.jp の検証状態（個別に検証している場合）
aws ses get-identity-verification-attributes \
  --identities "info@misesapo.co.jp" \
  --region "$REGION" \
  --query 'VerificationAttributes."info@misesapo.co.jp".VerificationStatus' \
  --output text

# アカウントの送信可否（サンドボックスかどうかはコンソールで確認）
aws ses get-account-sending-enabled --region "$REGION" --output json
```

- ドメイン／メールが `Success` でない場合は、SES の「検証」手順（DNS の TXT/CNAME 追加）を実施する必要があります。
- サンドボックスを解除するには、AWS サポートに「Production access」をリクエストします。

### 1.4 送信ログ（CloudWatch）で確認する

Lambda のログに `Registration form email sent to ...` や `Error sending ...` が出ているか確認します。送信処理で例外が出ていると、SES まで届いていません。

- **CloudWatch Logs** → 該当 Lambda のログストリームで `send_email` や `ses`、`Error` で検索。

## 2. 受信メールが届かない（外部 → @misesapo.co.jp / @misesapo.app）

お客様や取引先から **@misesapo.co.jp** や **@misesapo.app** あてに送ったメールが届かない場合です。受信は **EC2 や Lambda ではなく、メールホスト（レンタルサーバーや Google Workspace など）** で行われます。

### 2.1 どこのメールホストか確認する

ドメインごとに **MX レコード** で受信先が決まります。

```bash
# misesapo.co.jp の受信サーバー（MX）
dig MX misesapo.co.jp +short

# misesapo.app の受信サーバー（MX）
dig MX misesapo.app +short
```

- **Google Workspace**: `aspmx.l.google.com` などが並ぶ。
- **Xサーバー**: `*.xserver.jp` など。
- **ロリポップ**: `*.lolipop.jp` など。
- **ムームードメイン（お名前.com）**: `mail.onamae.com` など。

「届かない」ときは、この **MX の先のサービス**（契約しているレンタルサーバーや Google 管理コンソール）側を確認する必要があります。

### 2.2 受信側で確認すること

| 確認項目 | 内容 |
|----------|------|
| 契約・有効期限 | メールホストの契約が有効か、期限切れで止まっていないか |
| メールボックス容量 | 容量オーバーで受信拒否していないか |
| パスワード・ログイン | Webメールや IMAP でログインできるか（パスワード変更し忘れなど） |
| スパムフォルダ | 届いているがスパム扱いになっていないか |
| 転送・フィルタ | 転送先が間違っている、またはフィルタで捨てていないか |

### 2.3 送信テスト（外部から受信確認）

別のメールアドレス（Gmail など）から、`xxx@misesapo.co.jp` や `xxx@misesapo.app` に1通送ってみて、届くか・バウンスが返ってこないかを確認します。バウンスメールが返ってくる場合は、その内容（エラー文）で原因を絞り込めます。

---

## 3. info@misesapo.app に届かない場合（受信専用）

**届かない** = 外部から `info@misesapo.app` あてに送ったメールが受信できない、という前提で確認します。  
`misesapo.app` の受信は **Google Workspace** を使っている想定です。

### 3.1 MX が Google を向いているか（必須）

まず、ドメインの受信先が Google になっているか確認します。

```bash
dig MX misesapo.app +short
```

- **期待**: `aspmx.l.google.com.` など、Google の MX が並ぶ。
- **違う結果**（例: レンタルサーバーや別サービス）: そのサービス側で `info@` が作成されているか・有効かを確認する必要があります。  
  Google にしたい場合は、DNS の MX を Google Workspace 用に変更する必要があります。

### 3.2 「グループ」「ユーザー」「エイリアス」のどれか調べる

`info@misesapo.app` が **グループ / ユーザー / エイリアス** のどれかは、管理コンソールで次の順に確認するとわかります。

1. **https://admin.google.com/** にブラウザでアクセスし、管理者アカウントでログインする。

2. **まず「グループ」から探す**
   - 左メニューで **「ディレクトリ」** → **「グループ」** を開く。
   - 一覧に **`info@misesapo.app`** または **「info」** という名前のグループがあるか確認する。
   - **ある** → `info@` は **グループ**。そのグループをクリックし、下記「3.2a グループの場合」へ。
   - **ない** → 次に「ユーザー」を確認する。

3. **「ユーザー」で探す**
   - 左メニューで **「ディレクトリ」** → **「ユーザー」** を開く。
   - 一覧の「メール」列を見て、**メールアドレスが `info@misesapo.app` のユーザー**がいるか確認する。
   - **いる** → `info@` は **ユーザー**（info というユーザー）。そのユーザーをクリックし、下記「3.2b ユーザーの場合」へ。
   - **いない** → 「エイリアス」の可能性があるので、次へ。

4. **「エイリアス」かどうか調べる**
   - **「ユーザー」** 一覧で、**全員のユーザー**を順にクリックする。
   - 各ユーザーの詳細画面で **「ユーザー情報」** または **「メールエイリアス」** の欄を開く。
   - そこに **`info@misesapo.app`** が「エイリアス」として書いてあれば、そのユーザーが `info@` の実体。下記「3.2c エイリアスの場合」へ。
   - 誰のエイリアスにも無ければ、**`info@` はまだ作られていない**可能性がある（新規でユーザー／グループ／エイリアスを作る必要あり）。

**まとめ（どこにあったかで判断）**

| 見つかった場所 | info@ の種類 |
|----------------|----------------|
| ディレクトリ → グループ の一覧にあった | **グループ** |
| ディレクトリ → ユーザー の一覧で「メール」が info@misesapo.app だった | **ユーザー** |
| どれかのユーザー詳細の「メールエイリアス」に info@ があった | **エイリアス** |
| どこにも無かった | 未作成の可能性（新規作成が必要） |

---

**3.2a グループの場合**

- そのグループを開き、**「設定」**（または歯車アイコン）を開く。
- **「グループにメールを送信できるユーザー」** を確認する。
  - **「インターネット上の誰でも」** または **「組織内の全員」** → 外部からも送れる。届かない場合はスパム・転送・容量を確認。
  - **「組織内のユーザーのみ」** → **ここが原因**のことが多い。**「インターネット上の誰でも」** に変更すると、外部から送ったメールが届くようになる。
- 変更したら保存する。

**3.2b ユーザーの場合**

- そのユーザー（info）の詳細で、**ステータスが「有効」** か確認する。「停止」や「サスペンド」なら有効にする。
- メールボックス容量が満杯でないか、必要なら **「ユーザー」→ そのユーザー → ストレージ** などで確認する。

**3.2c エイリアスの場合**

- `info@` は「〇〇というユーザーの別名」なので、**そのユーザー（〇〇）** が有効か、停止・サスペンドになっていないか確認する。
- そのユーザーで Gmail にログインし、**転送設定** で `info@` あてのメールを捨てるルールになっていないか確認する。

### 3.3 届いていないのか、届いているが見えていないのか

- **Gmail（Web）**: `info@misesapo.app` でログインし、**「スパム」「ゴミ箱」「すべてのメール」** も含めて検索。
- **転送設定**: `info@` から別アドレスへ転送している場合、転送先で届いているか・転送が無効になっていないかを確認。
- **クライアント（Outlook 等）**: IMAP のフォルダを「すべて表示」にして、受信トレイ以外のフォルダも確認。

### 3.4 送信テスト

別のメール（Gmail や個人アドレスなど）から **info@misesapo.app** に1通送り、

- 届くか
- バウンス（送信エラー）が返ってこないか

を確認します。バウンスが返ってきたら、その本文のエラー内容で原因を絞り込めます（例: ユーザー不在、ドメイン拒否など）。

### 3.5 チェックリスト（info@misesapo.app）

- [ ] `dig MX misesapo.app +short` で Google の MX が出ている
- [ ] Google 管理コンソールで `info@misesapo.app` が「ユーザー」か「グループ」か「エイリアス」か把握した
- [ ] グループの場合「グループにメールを送信できるユーザー」で外部（インターネット）から送れる設定になっている
- [ ] ユーザー／グループが停止・サスペンドになっていない
- [ ] info@ で Web メールにログインし、スパム・ゴミ箱も含めて確認した
- [ ] 外部から送信テストして、届くか・バウンスが返らないか確認した

---

## 4. チェックリスト（どこで届いていないか）

- [ ] **システムから顧客へ送ったメール**が届かない  
  → 上記「1. 送信メール（SES）」を確認（ドメイン検証、サンドボックス、CloudWatch ログ）。

- [ ] **外部から info@misesapo.app へ送ったメール**が届かない  
  → 上記「3. info@misesapo.app に届かない場合」を確認（MX、Google のグループ/ユーザー設定、スパム・転送）。

- [ ] **その他 @misesapo.co.jp / @misesapo.app** が届かない  
  → 上記「2. 受信メール」を確認（`dig MX`、メールホストの契約・容量・ログイン・スパム）。

- [ ] **両方**届かない  
  → 送信と受信を別々に、上記 1 と 2／3 の順で確認。
